# Лобби — полный сценарий и тренер

Документ описывает, как работает лобби от создания до матча и как в него встроен **тренер** (в т.ч. исправление проблемы «тренер не видит лобби-бронь в расписании»).

---

## 1. Полный сценарий (как система должна работать)

```
1. Игрок создаёт лобби
   → POST /api/lobby/ { title, game_format, elo_min?, elo_max?, comment?, coach? }
   → Можно сразу указать coach_id (ID тренера) или добавить позже через PATCH.

2. Второй (и следующие) игроки вступают
   → POST /api/lobby/<id>/join/
   → Проверка ELO. Когда набралось нужное число игроков, статус → NEGOTIATING.

3. Согласование времени и корта
   → Любой участник: POST /api/lobby/<id>/proposals/ { court, scheduled_time, duration_minutes }
   → Участники голосуют: POST /api/lobby/<id>/proposals/<proposal_id>/vote/
   → Создатель принимает предложение: POST /api/lobby/<id>/proposals/<proposal_id>/accept/
   → Статус → READY, в лобби сохраняются court и scheduled_time.

4. Создатель назначает команды
   → POST /api/lobby/<id>/assign-teams/ { "teams": { "user_id": "A"|"B", ... } }

5. Создатель создаёт бронирование корта
   → POST /api/lobby/<id>/book/
   → На бэкенде создаётся Booking (court, user=создатель, start_time, end_time, participants, coach=lobby.coach).
   → Если в лобби был выбран тренер (lobby.coach), он записывается в бронь → тренер видит эту бронь в GET /api/bookings/coach/schedule/.

6. Каждый участник добавляет свои услуги (опционально) и оплачивает долю
   → POST /api/lobby/<id>/my-extras/ { services: [...] }
   → POST /api/lobby/<id>/pay-share/ { payment_method: "KASPI" }
   → Когда все оплатили, статус лобби → PAID, бронь → CONFIRMED.

7. Тренер видит лобби-матч в расписании
   → GET /api/bookings/coach/schedule/?from=...&to=...
   → В списке есть бронь с участниками (client_name, participants_names, players_for_match).

8. После игры тренер нажимает «Записать результат»
   → POST /api/gamification/matches/create/ { team_a, team_b, score, winner_team, court }
   → ELO обновляется, матч попадает в историю.

9. **Лобби без тренера:** после игры результат и ELO вводит **админ**. В приложении админ заходит в «Тренер» → «Матчи / Счёт» → «Загрузить брони» — в списке появляются и брони тренера, и **завершённые брони без тренера** (в т.ч. лобби). Выбирает бронь, распределяет игроков по командам A/B, указывает счёт и победителя → матч создаётся, ELO начисляется. API: `GET /api/bookings/completed-without-coach/` (только ADMIN).

10. Лобби можно считать закрытым по факту (статус уже PAID); при необходимости — закрытие через UI/кнопку.
```

---

## 2. В чём была проблема (и что исправлено)

**Жалоба фронтенда:** «Тренер не видит лобби-матч в расписании», «нет API для полного утверждения лобби», «выбор тренера не сохраняется».

**Причина:**

- В модели **Lobby** не было поля **тренер**.
- При создании брони из лобби (`POST /api/lobby/<id>/book/`) создавалась запись **Booking** **без** `coach`.
- Расписание тренера строится как `Booking.objects.filter(coach=user)` → брони из лобби (с пустым `coach`) **не попадали** в расписание тренера.

**Что сделано:**

1. В модель **Lobby** добавлено поле **`coach`** (FK на User, null=True).
2. При создании брони из лобби в неё подставляется **`booking.coach = lobby.coach`**. Если в лобби выбран тренер, он сохраняется в брони и тренер видит эту бронь в своём расписании.
3. **Тренера можно задать при создании лобби** и **поменять до создания брони**:
   - **POST /api/lobby/** — в теле можно передать `coach` (ID пользователя-тренера).
   - **PATCH /api/lobby/<id>/** — только создатель может изменить `coach` и `comment`, и только пока статус не `BOOKED`/`PAID`.

## 3. API по лобби и тренеру

### Создание лобби (с тренером или без)

```http
POST /api/lobby/
Authorization: Bearer <token>
Content-Type: application/json

{
  "title": "Вечерняя игра",
  "game_format": "DOUBLE",
  "elo_min": 1000,
  "elo_max": 1400,
  "comment": "С тренером",
  "coach": 5
}
```

- **coach** — опционально, ID пользователя с ролью тренера (COACH_PADEL / COACH_FITNESS). Список тренеров: `GET /api/auth/coaches/`.

### Обновление тренера или комментария (до создания брони)

```http
PATCH /api/lobby/<id>/
Authorization: Bearer <token>
Content-Type: application/json

{
  "coach": 5,
  "comment": "Обновлённый комментарий"
}
```

- Доступно **только создателю** лобби.
- Разрешено только пока статус лобби **не** `BOOKED` и **не** `PAID`.

### Ответы лобби (GET /api/lobby/, GET /api/lobby/<id>/)

В объекте лобби теперь есть:

- **coach** — ID тренера или `null`.
- **coach_name** — имя тренера (для отображения).

Пример:

```json
{
  "id": 10,
  "creator": 42,
  "creator_name": "Азамат",
  "title": "Вечерняя игра",
  "game_format": "DOUBLE",
  "status": "NEGOTIATING",
  "coach": 5,
  "coach_name": "Алексей Тренер",
  "court": null,
  "scheduled_time": null,
  ...
}
```

### Расписание тренера (приложение тренера)

```http
GET /api/bookings/coach/schedule/?from=2026-03-01&to=2026-03-31
Authorization: Bearer <coach_token>
```

В ответе — все брони, где **этот пользователь назначен тренером**, в том числе брони, созданные из лобби с выбранным этим тренером. В каждой брони есть поля для отображения клиента, участников и подстановки в «Записать результат» (`players_for_match` и т.д.).

## 4. Где что хранится

| Что | Где хранится |
|-----|------------------|
| Выбор тренера для лобби | **Lobby.coach** (FK User) |
| Созданная из лобби бронь | **Booking** (bookings_booking), связь **Lobby.booking** (FK) |
| Тренер в брони | **Booking.coach** — при создании брони из лобби копируется из **Lobby.coach** |
| Расписание тренера | Выборка **Booking.objects.filter(coach=request.user)** в `GET /api/bookings/coach/schedule/` |

Лобби после «Создать бронь» никуда отдельно не «подтверждается» — достаточно того, что создаётся одна бронь, к ней привязывается лобби (`lobby.booking_id`), а тренер из лобби записывается в эту бронь. Этого достаточно, чтобы тренер видел лобби-матч в своём расписании и мог записать результат.

## 5. Рекомендации для мобилки (клиент и тренер)

**Клиентское приложение:**

- При создании лобби показывать выбор тренера (список из `GET /api/auth/coaches/`), передавать **coach** в **POST /api/lobby/**.
- На экране лобби (до перехода в READY/создания брони) дать возможность сменить тренера через **PATCH /api/lobby/<id>/** с полем **coach**.
- После «Создать бронь» не требовать отдельного «полного утверждения» лобби — достаточно успешного ответа от **POST /api/lobby/<id>/book/**.

**Приложение тренера:**

- Расписание по-прежнему загружать через **GET /api/bookings/coach/schedule/** — в нём уже будут и обычные брони, и брони из лобби, где выбран этот тренер.
- После игры использовать **POST /api/gamification/matches/create/** с данными из брони (участники, корт и т.д.).

## 6. Что происходит после завершения матча (время брони прошло)

**Бронь (Booking):**

- Когда время брони истекло (`end_time` в прошлом), бронь переводится в статус **COMPLETED**:
  - автоматически при следующем запросе списка/детали броней (в представлениях вызывается `complete_past_bookings()`);
  - либо вручную: `python manage.py complete_past_bookings`.
- Отдельного «завершения лобби» по времени нет: **лобби остаётся в статусе PAID**.

**Результат матча и ELO:**

- Обновление рейтинга **не** происходит само по себе. Чтобы засчитать результат и изменить ELO:
  - тренер (или админ) создаёт матч: **POST /api/gamification/matches/create/** с полями `team_a`, `team_b`, `score`, `winner_team`, `court` и т.д.;
  - тогда начисляется/списывается ELO (логика в `gamification`).
- Связь «лобби → матч» не автоматическая: матч создаётся отдельно (по расписанию тренера или из CRM). При желании можно связать матч с бронь/лобби в своей логике (поля в Match или расширение API).

**Итого:** после окончания времени — бронь становится COMPLETED; лобби остаётся PAID; результат и ELO учитываются только после создания матча в геймификации.
