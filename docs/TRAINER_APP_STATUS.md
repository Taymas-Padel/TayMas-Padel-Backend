# Приложение для тренеров — что готово и чего не хватает

**Все эндпоинты и форматы запросов/ответов для тренеров описаны в основном документе:** `API_MOBILE.md` (разделы 5.7, 5.8, 5.10, 8.1–8.3, 14.4, раздел 20 «Client app vs Trainer app»). Здесь — только сводка по готовности и ограничениям бэкенда.

---

## ✅ Что уже готово

### 1. Расписание (кто забронировал время с тренером)

| Эндпоинт | Описание |
|----------|----------|
| `GET /api/bookings/coach/schedule/?from=YYYY-MM-DD&to=YYYY-MM-DD` | Расписание броней, где тренер назначен. Без параметров — с сегодня по +14 дней. |

**Ответ:** список броней с полями:
- `id`, `court`, `court_name`, `user`, `client_name`, `start_time`, `end_time`, `duration_hours`
- `coach`, `coach_name`, `participants_names`
- **`players_for_match`** — массив `[{id, name}, ...]` (клиент + участники) для удобного выбора игроков в матч
- `services`, `price`, `status`, `is_paid`, `created_at`

Тренер видит: когда, какой корт, кто клиент, кто участники — всё есть.

---

### 2. Детали брони и отмена

| Эндпоинт | Описание |
|----------|----------|
| `GET /api/bookings/{id}/` | Детали брони (тренер видит брони, где он coach). |
| `POST /api/bookings/{id}/cancel/` | Отмена брони (разрешена владельцу, ресепшн, админу и **тренеру этой брони**). |

Правило отмены: за 24 ч до начала (или по настройке клуба); тренер может отменить свою бронь так же, как клиент.

---

### 3. Геймификация: создание матча и ELO

| Эндпоинт | Описание |
|----------|----------|
| `POST /api/gamification/matches/create/` | Создать матч. **Только тренер или ADMIN.** |
| `GET /api/gamification/matches/` | История матчей (тренер видит свои как игрок; с `?all=true` — все матчи). |

**Создание матча (тело запроса):**
```json
{
  "team_a": [42, 15],
  "team_b": [22, 33],
  "score": "6:4",
  "winner_team": "A",
  "court": 1
}
```

- `judge` проставляется автоматически (текущий пользователь — тренер).
- После сохранения автоматически пересчитывается ELO: победители +25, проигравшие −25 (минимум 0).
- `elo_changes` сохраняются в матче и отдаются в ответе.

То есть: тренер может выставить результат и команды, ELO начисляется — **это готово.**

---

### 4. Вход и профиль

- Тренеры входят через **тот же мобильный вход по SMS**: `POST /api/auth/mobile/login/` (роль `COACH_PADEL` или `COACH_FITNESS`).
- Профиль: `GET /api/auth/users/me/`, `PATCH /api/auth/users/me/`.
- Статистика: `GET /api/auth/me/stats/`, лига: `GET /api/auth/me/league/`.

---

### 5. Персональные тренировки (зал)

| Эндпоинт | Описание |
|----------|----------|
| `GET /api/gym/personal-training/` | Тренер видит тренировки, где он coach. |
| `POST /api/gym/personal-training/` | Создать тренировку (тренер может создать на себя). |
| `GET/PATCH/DELETE /api/gym/personal-training/{id}/` | Просмотр/редактирование/удаление. |

## ❌ Чего нет (что можно добавить)

### 1. Счёт командам за матч (оплата за судейство/игру)

Сейчас при создании матча:
- сохраняется результат и команды;
- начисляется ELO;
- **не создаётся никакой платёж/счёт.**

Если по ТЗ тренер должен **выставлять счёт командам** (оплата за судейство, аренду или что-то ещё по геймификации), в API этого нет.

**Что можно сделать:**
- Вариант A: при создании матча опционально передавать `amount`, `payment_method` и делить сумму на команды (или одну сторону) и создавать транзакции (например, тип `MATCH_FEE` или привязка к брони).
- Вариант B: отдельный эндпоинт `POST /api/gamification/matches/{id}/charge/` — «выставить счёт за матч» (сумма, кому: команда A / команда B / обе, способ оплаты), создание транзакций и, при необходимости, запись в `Transaction` с привязкой к матчу (если добавить в модель `Match` поле или связь с платежами).

Имеет смысл уточнить в ТЗ: кто платит (обе команды поровну, одна команда, клиент брони), за что именно (судейство, аренда, что-то ещё), и тогда спроектировать один из вариантов.

---

### 2. Связь матча с бронированием

Сейчас:
- модель `Match` **не связана** с `Booking`;
- тренер вручную выбирает `team_a` и `team_b` (список игроков может взять из брони через `players_for_match` в ответе расписания).

Если нужно:
- «создать матч по брони» — подставить команды из брони и привязать матч к брони;
- показывать в карточке матча «по брони #101»;
- автоматически учитывать оплату за корт по брони, а счёт тренера — только за судейство,

то нужно:
- добавить в модель `Match` опциональное поле `booking = ForeignKey(Booking, null=True, blank=True)` (и миграцию);
- в API: опциональный параметр `booking_id` при создании матча (если передан — подтягивать игроков из брони и сохранять связь).

Это не обязательно для «видеть расписание и выставлять результат», но удобно для сценария «матч по этой брони».

---

### 3. Дополнительно (по желанию)

- **Уведомления тренеру:** при новой брони с тренером уже можно слать пуш (если на бэкенде вызывается `send_notification` при создании брони с coach) — стоит проверить, что тренер получает уведомления и что в приложении тренера есть экран уведомлений (`/api/notifications/`).
- **Фильтр «только мои матчи»:** тренер уже видит свои матчи как участник; с `?all=true` видит все. При необходимости можно добавить явный фильтр «только матчи, где я судья» (`judge=request.user`), если это нужно в UI.

## Итог

| Требование | Статус |
|------------|--------|
| Тренер видит своё расписание (кто забронировал с ним) | ✅ Готово: `GET /api/bookings/coach/schedule/` |
| Видно, кто забронировал (клиент + участники) | ✅ В ответе есть `client_name`, `participants_names`, `players_for_match` |
| Выставить результат матча командам (геймификация) | ✅ Готово: `POST /api/gamification/matches/create/`, ELO считается |
| Выставить **счёт/оплату** командам за матч | ❌ Нет в API — нужно проектировать и добавить |
| Связь матча с бронированием | ❌ Нет — при необходимости добавить `booking_id` и логику «матч по брони» |

Для **приложения тренеров** достаточно готово: расписание, детали брони, отмена, создание матча и ELO. Не хватает только **выставления счёта за матч** (и при желании — привязки матча к брони и донастройки уведомлений).
