# API: ожидаемое число участников для цены тренера (обычная бронь)

Чтобы корректно считать цену тренера, когда в приложении не указаны друзья, но на тренировку по факту придут 3–4 человека, в обычной брони добавлено **отдельное поле** — ожидаемое число участников.

---

## 1. Поле `coach_expected_participants`

| Параметр | Тип | Обязательное | Описание |
|----------|-----|:------------:|----------|
| `coach_expected_participants` | integer | нет | Сколько человек **будет на тренировке** (1, 2, 3 или 4). Используется **только для расчёта цены тренера**: 1–2 → тариф «1–2 игрока», 3–4 → тариф «3–4 игрока». Если не передано — считается по `1 + len(friends_ids)`. |

- Допустимые значения: **1, 2, 3, 4**.
- Имеет смысл передавать только при выбранном тренере (`coach` / `coach_id`).
- В лобби это поле **не используется** — там число участников известно из состава лобби.

---

## 2. Создание брони

**POST /api/bookings/create/**

**Тело запроса (фрагмент):**

```json
{
  "court": 1,
  "start_time": "2026-04-15T14:00:00Z",
  "duration": 90,
  "coach": 3,
  "coach_expected_participants": 4,
  "friends_ids": [],
  "payment_method": "KASPI"
}
```

| Поле | Когда передавать |
|------|------------------|
| `coach_expected_participants` | По желанию. Если передан и указан `coach` — цена тренера считается по этому числу (1–2 или 3–4). Если не передан — используется `1 + количество друзей в friends_ids`. |

**Примеры:**

- Бронь с тренером, друзей в приложении нет, но придут 4 человека → передать `coach_expected_participants: 4` → применится тариф «3–4».
- Бронь с тренером и 2 друзьями в `friends_ids` → можно не передавать поле → будет 3 участника → тариф «3–4».
- Бронь с тренером, в приложении 1 друг, по факту будет только он и клиент → можно передать `coach_expected_participants: 2` или не передавать (и так получится 2).

---

## 3. Превью стоимости (price-preview)

**POST /api/bookings/price-preview/**

**Тело запроса (фрагмент):**

```json
{
  "court_id": 1,
  "start_time": "2026-04-15T14:00:00Z",
  "duration": 90,
  "coach_id": 3,
  "coach_expected_participants": 4,
  "service_ids": [],
  "friends_ids": []
}
```

| Поле | Когда передавать |
|------|------------------|
| `coach_expected_participants` | По желанию. Если передан и указан `coach_id` — в расчёте стоимости тренера используется это число (1–2 или 3–4). Иначе — `1 + len(friends_ids)`. |

Чтобы превью совпадало с итогом при создании брони, в оба запроса (price-preview и create) нужно передавать одинаковое значение `coach_expected_participants` при выбранном тренере.

---

## 4. Что сделать на фронте

1. **Форма обычной брони (с тренером)**  
   Когда выбран тренер, показать вопрос: **«Сколько человек будет на тренировке?»** с вариантами:
   - **1–2 человека** (отправлять значение **2** для тарифа 1–2)
   - **3–4 человека** (отправлять значение **4** для тарифа 3–4)

   Можно хранить в селекте значения `2` и `4` и передавать их в API как `coach_expected_participants`. Допустимы и значения 1, 2, 3, 4: бэкенд подставляет их в расчёт цены тренера (1–2 → тариф 1–2, 3–4 → тариф 3–4).

2. **Запрос создания брони**  
   В body добавлять `coach_expected_participants` только если выбран тренер (и пользователь выбрал количество). Иначе не передавать — тогда будет использоваться `1 + friends_ids.length`.

3. **Запрос price-preview**  
   В body добавлять то же поле `coach_expected_participants` при выбранном тренере, чтобы превью и итоговая сумма совпадали.

4. **Лобби**  
   В лобби это поле не нужно: число участников берётся из состава лобби (2 или 4).

---

## 5. Кратко

- Новое поле: **`coach_expected_participants`** (integer 1–4), необязательное.
- Используется в **POST /api/bookings/create/** и **POST /api/bookings/price-preview/** при выбранном тренере.
- Задаёт, по какому тарифу считать тренера (1–2 или 3–4), даже если в приложении не указаны друзья.
- На фронте: при выборе тренера показать «Сколько человек будет на тренировке?» (1–2 / 3–4) и передавать выбранное значение в оба запроса.
